# AI-First 工程实施手册

> 把工程组织从「AI 辅助」重构为「AI 主建、人做方向与判断」的完整蓝图。
> 适配场景:GitHub + GitHub Actions、多语言混合栈、<15 人团队、创始人/CTO 亲自推动。

---

## 0. 一句话定调

**AI-First 不等于用 AI。** 大多数公司把 AI 螺丝在现有流程上——工程师开 Cursor、PM 用 ChatGPT 写文档、QA 试试 AI 生成测试——流程不变,效率涨 10~20%,结构毫无改变。这叫 **AI-assisted(AI 辅助)**。

**AI-First** 是:在「AI 是主要建造者」这一假设下,重新设计你的**流程、架构、组织**。你不再问"AI 怎么帮我们的工程师?",而是问"如何重组一切,让 AI 来建,工程师只提供方向与判断?"。两者的差距是**乘性的**,不是加性的。

本手册就是把这个重构拆成可执行的阶段、角色、检查清单和决策标准。

---

## 1. 先想清楚:你是不是真的要做这件事

在动手前,用三个问题自测,任意一个为「是」就说明你已经在被瓶颈拖住:

1. **你的产品规划周期,是否长于你的构建周期?** 如果想一个功能要几周、而 agent 实现只要几小时,规划就是瓶颈。
2. **你的测试/验证速度,是否跟不上实现速度?** 如果 agent 2 小时出功能、QA 3 天测完角落用例,你只是把瓶颈往下游挪了 10 英尺。
3. **你是否在靠堆人追赶产能?** 竞品 100 倍的人做同样的事,你招不过来,只能「重新设计出」产能。

三个系统必须都让 AI 贯穿:**怎么设计产品、怎么实现产品、怎么测试产品**。任何一个仍是人工,就会卡住整条流水线。

> ⚠️ 警告:这不是 vibe coding。开 Cursor 一直 prompt 到能跑、commit、重复——那只产出原型。**生产系统要稳定、可靠、安全**。你需要一个**在 AI 写代码时仍能保证这些属性的系统**。系统是你建的,prompt 是一次性的、可丢弃的。

## 2. 三大瓶颈与对策(诊断)

| 瓶颈 | 旧世界症状 | 对策 |
|---|---|---|
| **产品管理** | PM 花几周研究、设计、写规格;agent 2 小时实现 | PM 进化为**产品型架构师**,以迭代速度工作;用「原型-上线-测试-迭代」循环替代「委员会评审规格文档」 |
| **QA** | agent 2 小时出功能,QA 3 天测角落用例 | 用 **AI 构建的测试平台测 AI 写的代码**;验证速度必须等于实现速度 |
| **人头** | 竞品 100 倍人力 | 重新设计而非招聘;用 architect + operator 两类角色 + agent 放大产能 |

记住核心判断句:**「花几个月想一件事、然后 2 小时把它建出来」是不合理的。**

## 3. 第一个大动作:统一架构(monorepo)

> 这是整套转型的**前置条件**,不做这步,后面全是空中楼阁。

### 为什么必须统一

旧架构常散落在多个独立系统里,一个改动要碰 3~4 个仓库。**从人类工程师视角,可管理;从 agent 视角,不透明**:

- agent 看不到全局
- agent 无法推理跨服务影响
- agent 无法在本地跑集成测试

这正是 harness engineering 原则的实践:**你把系统中越多的部分,变成 agent 能检视、校验、修改的形态,你的杠杆就越大。碎片化的代码库对 agent 不可见;统一的代码库可被读懂(legible)。**

### 怎么做(给小团队的节奏)

CREAO 的做法是:**一周设计新系统**(规划阶段、实现阶段、测试阶段、集成测试阶段四个阶段),**再一周用 agent 重构整个代码库**。

对 <15 人团队的渐进式建议(不要一刀切停摆两周):

1. 建一个新的 monorepo 骨架(工具:Nx / Turborepo / Bazel / Go workspaces / pnpm workspaces,按你的语言选)。
2. **先迁移耦合最紧的 2~3 个服务**,把跨服务的接口、类型、契约放进共享包。
3. 在根目录放 [`AGENTS.md`](../AGENTS.md):写清目录结构、各服务职责、如何本地起全栈、如何跑集成测试、编码规范、禁区。这是 agent 的"地图"。
4. 保证 **一条命令能本地起全栈 + 一条命令能跑集成测试**。agent 无法在本地复现,就无法自我验证。
5. 其余服务在后续 sprint 逐步并入。

**判断标准(Definition of Legible):** 一个全新的 agent,只读 monorepo + AGENTS.md,能否独立完成「改一个跨两个服务的功能并让集成测试通过」?能,就算统一到位了。

## 4. 技术栈与各部件职责

> 工具可替换,**职责不可省**。每个部件的存在理由都是同一句:**如果 AI 读不到信号,它就无法诊断问题。**

### 4.1 基础设施 + 可观测(文章:AWS + CloudWatch)

- 自动扩缩容的容器服务 + **熔断式回滚**:部署后指标恶化,系统自动回退。
- 可观测后端是**中枢神经**:全服务结构化日志、>25 条告警、自定义指标,被自动化工作流每日查询。
- 原则:**每一块基础设施都暴露结构化、可查询的信号。**
- 替代:GCP Cloud Monitoring、Prometheus + Loki + Grafana + Alertmanager、阿里云 SLS。

### 4.2 CI/CD(文章:GitHub Actions)— 六阶段流水线

每个代码改动都经过六个阶段,**没有任何阶段可选,没有人工绕过**:

```
Verify CI → Build & Deploy Dev → Test Dev → Deploy Prod → Test Prod → Release
```

每个 PR 的 **CI 门禁** 强制:类型检查、lint、单元测试、集成测试、Docker 构建、Playwright E2E、环境一致性检查。**流水线是确定性的,所以 agent 能预测结果、能推理失败。**

> 落地文件:[`.github/workflows/ci.yml`](../.github/workflows/ci.yml)、[`deploy.yml`](../.github/workflows/deploy.yml)。

### 4.3 AI 代码评审(文章:Claude Opus)— 并行三趟

每个 PR 触发三趟并行 AI 评审:

1. **代码质量**:逻辑错误、性能问题、可维护性。
2. **安全**:漏洞扫描、认证边界检查、注入风险。
3. **依赖扫描**:供应链风险、版本冲突、许可证问题。

这些是**评审门禁,不是建议**。与人类评审并行,在高吞吐下抓住人会漏的东西。**当你一天部署 8 次,没有人类评审者能对每个 PR 持续保持注意力。**

工程师还可在任意 GitHub issue/PR 里 `@` 提到 agent,请它做实现计划、调试、代码分析——agent 看得到整个 monorepo,上下文跨对话延续。

> 落地文件:[`.github/workflows/ai-review.yml`](../.github/workflows/ai-review.yml) + [`prompts/`](../prompts/)。

### 4.4 自愈反馈环(★ 中心枢纽)

这是整套系统的**核心**,一个每日自动循环:

- **每天 09:00 UTC**:健康工作流自动跑。模型查可观测后端,分析全服务错误模式,生成**执行级健康摘要**推给团队(Teams/Slack)。**没人需要去要它。**
- **一小时后(10:00 UTC)**:triage 引擎跑。它把生产错误(可观测后端 + Sentry)**聚类**,对每个簇按**九个严重度维度打分**,自动在 Linear 生成调查工单。每张工单含:样本日志、受影响用户、受影响端点、建议的调查路径。
- **去重**:已有 open 工单覆盖同一错误模式 → 更新它;之前 closed 的工单复现 → 检测到回归并**重开**。
- **闭环**:工程师推修复 → 同一条流水线(三趟 AI 评审 + CI + 六阶段部署)处理 → 部署后 triage 引擎**复检**可观测后端 → 原错误消失则**自动关闭** Linear 工单。

> 每个工具只做一个阶段,没有哪个工具试图全包。
> 落地文件:[`.github/workflows/daily-health.yml`](../.github/workflows/daily-health.yml)、[`triage.yml`](../.github/workflows/triage.yml)、[`scripts/`](../scripts/)。

### 4.5 特性开关与支撑栈

- **特性开关(文章:Statsig)**:每个功能都藏在开关后。灰度模式:先对团队开 → 按百分比逐步放量 → 全量发布或杀掉。**kill switch 即时关闭功能,无需部署。** 指标恶化几小时内拉掉,坏功能当天上当天死。A/B 测试走同一套系统。
- **PR 分支管理(文章:Graphite)**:合并队列 rebase 到 main、重跑 CI、只在绿了才合;栈式 PR 支持高吞吐下的增量评审。替代:GitHub Merge Queue。
- **异常上报(文章:Sentry)**:全服务结构化异常,由 triage 引擎与可观测日志合并做跨工具关联。
- **人面工单层(文章:Linear)**:自动建单含严重度分数、样本日志、建议调查;去重防噪;复检自动关闭。

> 落地文件:[`flags/feature-flags.ts`](../flags/feature-flags.ts)。

## 5. 一个功能/一个 bug 如何流动(标准作业流程 SOP)

两条路径**用同一条流水线,一套标准**。

### 5.1 新功能路径

1. **架构师**把任务定义为**结构化 prompt**:含代码库上下文、目标、约束。(模板:[`prompts/architect-task.md`](../prompts/architect-task.md))
2. **agent** 分解任务、规划实现、写代码、**生成自己的测试**。
3. **开 PR**。三趟 Claude 评审。**人类评审者只看战略风险,不看逐行正确性。**
4. **CI** 校验:类型、lint、单测、集成、E2E。
5. **合并队列** rebase、重跑 CI、绿了才合。
6. **六阶段部署** 经 dev/prod,每阶段带测试。
7. **特性开关** 对团队开 → 按百分比灰度 → 监控指标。
8. **kill switch** 待命;严重问题**熔断自动回滚**。

### 5.2 Bug 修复路径

1. 可观测后端 + Sentry **检测错误**。
2. triage 引擎**打分**严重度,建带完整调查上下文的 Linear 工单。
3. 工程师调查——**AI 已经做完了诊断**,工程师**验证并推修复**。
4. 同样的评审、CI、部署、监控流水线。
5. triage 引擎**复检**;已解决则工单**自动关闭**。

> 一句话:**AI 会把 PR 提好,人类只需要审查有没有风险。**

## 6. 新的工程组织:两类工程师

转型后只有两类工程师角色。

### 6.1 架构师(Architect)— 1~2 人

- 设计**教 AI 如何工作的标准作业流程(SOP)**;建测试基础设施、集成系统、triage 系统;定架构与系统边界;**定义对 agent 而言"好"长什么样**。
- **需要极强的批判性思维。你批判 AI,而不是跟随它。** agent 给方案,架构师找漏洞:漏了哪些失败模式?越过了哪条安全边界?在累积什么技术债?
- 文章作者(物理学 PhD)总结:PhD 教他最有用的是**质疑假设、压力测试论证、寻找缺失项**。**批判 AI 的能力,将比产出代码的能力更值钱。**
- 这是**最难招的角色**。

### 6.2 操作员(Operator)— 其余所有人

- 工作依然重要,只是结构变了。**AI 给人派活**:triage 系统发现 bug、建单、给出诊断、指派给对的人;人调查、验证、批准修复;**AI 提 PR,人审查有没有风险**。
- 任务类型:bug 调查、UI 打磨、CSS 改进、PR 评审、验证。需要技能与注意力,但**不需要旧模型要求的那种架构推理**。

### 6.3 谁适应得最快(管理预期用)

文章观察(非价值判断,仅描述现象):**初级工程师比资深工程师适应更快**。初级没有十年的习惯要忘掉,觉得被赋能;资深的强传统功底反而最难——"我两个月的工作 AI 一小时干完"是years 攒下的稀缺技能后很难接受的事。**转型期,适应力比累积技能更重要。**

## 7. 人的一面(转型成本,必须正视)

- **管理坍缩**:作者从 60% 时间管人(对齐、开会、反馈、辅导)降到 <10%。因为系统只需要 1~2 个架构师,CTO 得先自己做架构、自己建。从"管理"转向"建造"。更累(常 9am–3am),但享受建造而非对齐。
- **争吵变少、关系变好**:大量"对齐会、辩论权衡、技术分歧"是传统模型的必需,也很耗。现在这些能被系统轻松完成的工作不再需要争,团队聊别的、关系更好。
- **不确定感是真实的**:CTO 不再每天找人,有人会慌——"CTO 不找我意味着什么?我在这个新世界的价值是什么?" 合理的担忧,**没有干净的答案**。有人花在"辩论 AI 能否做我的活"上的时间比做活还多。
- **一条原则**:**不会因为引入了生产 bug 就开除工程师——而是改进评审、加强测试、加护栏。对 AI 一视同仁。AI 犯错,就建更好的校验、更清晰的约束、更强的可观测。**

## 8. 超越工程:让每个职能都跑在 agent 速度

如果工程几小时出功能,但市场部要一周才发公告,**市场部就是瓶颈**;产品团队还在跑月度规划,**规划就是瓶颈**。把 AI-native 推进每个职能:

- 产品发布说明:从 changelog 与功能描述 AI 生成。
- 功能介绍视频:AI 生成动态图形。
- 社媒日更:AI 编排自动发布。
- 健康报告与分析摘要:从可观测后端与生产库 AI 生成。

**工程、产品、市场、增长跑在同一条 AI-native 工作流里。一个职能跑 agent 速度、另一个跑人类速度,人类速度那个就拖住一切。**

## 9. <15 人小团队迁移路线图(4 周渐进,不停摆)

> 原则:**先建测试脚手架,再放大 agent。快 AI 没有快校验 = 高速堆积技术债。**

### 第 0 周 — 决定与对齐(0.5 周)

- CTO 内部宣布:我们要重构流程,不是加工具。讲清成本(真空期、不确定感)。
- 选定 1 名架构师(很可能就是 CTO 本人先上)。
- 用第 1 节三问做一次现状诊断,记录当前各环节耗时基线(用于事后对比)。

### 第 1 周 — 让代码库对 agent 可见

- 建 monorepo 骨架,先并入耦合最紧的 2~3 个服务。
- 写根 `AGENTS.md`:目录、服务职责、本地起全栈命令、跑集成测试命令、规范、禁区。
- 落地 [`ci.yml`](../.github/workflows/ci.yml):至少 类型检查 + lint + 单测 跑起来,**门禁强制、不可绕过**。
- 验收:新 agent 只读 monorepo + AGENTS.md 能完成一个跨服务小改动并过 CI。

### 第 2 周 — 建验证脚手架(最关键的一周)

- 补集成测试 + E2E(Playwright 或等价),纳入 CI 门禁。
- 落地 [`ai-review.yml`](../.github/workflows/ai-review.yml) 三趟评审 + [`prompts/`](../prompts/)。
- 落地 [`deploy.yml`](../.github/workflows/deploy.yml) 六阶段 + 熔断回滚(先 dev,再接 prod)。
- 验收:一个由 agent 写的 PR,能在无人工逐行审的情况下,被三趟评审 + CI 拦住至少一类典型问题。

### 第 3 周 — 接自愈环 + 特性开关

- 接可观测后端结构化日志 + 告警 + Sentry。
- 落地 [`daily-health.yml`](../.github/workflows/daily-health.yml) + [`triage.yml`](../.github/workflows/triage.yml) + [`scripts/`](../scripts/),打通 → 工单层(Linear/Jira/GitHub Issues)。
- 接 [`flags/feature-flags.ts`](../flags/feature-flags.ts):每个新功能藏开关后,配 kill switch 与灰度。
- 验收:跑通一次完整闭环——制造一个错误 → 自动建单含诊断 → agent 修 → 流水线部署 → 自动复检关单。

### 第 4 周 — 放大与外溢

- 把剩余服务并入 monorepo。
- 提高每日部署频次(目标:3~8 次/天),观察 A/B 与指标。
- 把 AI-native 推到产品/市场/增长(第 8 节)。
- 复盘:对比第 0 周基线,看用户参与度、支付转化、发布频率。

## 10. 成功度量(怎么知道做对了)

文章给的实测结果可作为目标参照:

- **14 天内平均每天 3~8 次生产部署**(旧模型同期可能一次正式发布都没有)。
- 坏功能**当天拉掉**,新功能**当天上线**,A/B **实时验证影响**。
- 反直觉但真实:**用户参与度↑、支付转化↑**。不是用质量换速度——**反馈环更紧,你日更比月更学到更多**,结果反而更好。

给自己定的领先/滞后指标建议:

- 领先:PR 从开到合的中位时长、CI 通过率、agent 自动生成测试的覆盖率、triage 自动建单/自动关单比例。
- 滞后:每日部署次数、生产事故 MTTR、坏功能从上线到杀掉的时长、核心业务指标(参与度/转化)。

## 11. 给不同读者的行动清单

**给工程师:** 你的价值从"代码产出"转向"决策质量"。写代码快越来越不值钱,评估/批判/指挥 AI 越来越值钱。**产品 sense/品味很重要**——看到生成的 UI 能在用户开口前就知道它不对吗?看架构方案能看出 agent 漏掉的失败模式吗?训练批判性思维:评估论证、找缺口、质疑假设、知道好设计长什么样——这些会复利。

**给 CTO/创始人:** ①如果 PM 流程比构建还慢,从那里开始。②**先建测试脚手架,再放大 agent。**③从一个架构师起步,系统跑通后再把其他人 onboard 成 operator。④把 AI-native 推到每个职能。⑤预期阻力,有人会推回。

**给行业:** OpenAI、Anthropic 与多个独立团队收敛到同一组原则——结构化上下文、专用 agent、持久记忆、执行循环。**模型能力是驱动这一切的时钟**:作者把 CREAO 的整场转变归因于"最近两个月"的模型进步,下一代模型会进一步加速。一人公司会变得常见——一个架构师 + agents 能做 100 人的活时,很多公司不需要第二名员工。
