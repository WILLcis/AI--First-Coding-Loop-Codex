# 多模型适配(v2.1)

> harness v2.1 起,**所有 LLM 调用都走 `ModelAdapter` 单点**——切厂商只改两个 env,业务代码零改动。
> 用 OpenAI 给项目 A、用 DeepSeek 给项目 B、用 Qwen 给项目 C 同时跑,毫无问题。
>
> v2.6.3 起,默认 PR AI Review workflow 已改走 Codex CLI + `OPENAI_API_KEY` / `CODEX_ACCESS_TOKEN`。本文保留给 legacy 云 LLM 评审、triage、health、goal_loop 等仍直接调用 `ModelAdapter` 的脚本。

---

## 1. 一句话切厂商

```bash
export LLM_PROVIDER=deepseek           # anthropic | openai | deepseek | qwen | kimi | glm | baichuan | siliconflow | custom
export LLM_API_KEY=<你的 key>
# 完成。triage、health、ai_review、goal_loop 等脚本自动走新厂商。
```

在 **GitHub Actions** 里就是:
- 设 vars **`LLM_PROVIDER`** = `deepseek`
- 设 secrets **`LLM_API_KEY`** = `sk-...`
- (可选) 设 vars **`LLM_BASE_URL`** 用于私有部署或预设里没列的厂商
- (可选) 设 vars **`LLM_MODEL_VERIFIER_SECURITY`** 等用于 per-role 覆盖

## 3. per-role 模型覆盖(把强模型只留给关键 role)

脚本调用通过环境变量做 **per-role** 覆盖,实现更细的成本/质量分层。
Codex custom agents 位于 `.codex/agents/`,默认继承 Codex 会话模型;它们主要负责角色边界与 `model_reasoning_effort`。

变量命名规则:`LLM_MODEL_<UPPER_ROLE_WITH_UNDERSCORES>`

| role | env 覆盖键 |
|---|---|
| explorer | `LLM_MODEL_EXPLORER` |
| implementer | `LLM_MODEL_IMPLEMENTER` |
| verifier-quality | `LLM_MODEL_VERIFIER_QUALITY` |
| verifier-security | `LLM_MODEL_VERIFIER_SECURITY` |
| verifier-dependency | `LLM_MODEL_VERIFIER_DEPENDENCY` |
| triage-scorer | `LLM_MODEL_TRIAGE_SCORER` |
| checker | `LLM_MODEL_CHECKER` |

**示例:用 DeepSeek 做主力,security 趟单独升到 deepseek-reasoner**

```bash
export LLM_PROVIDER=deepseek
export LLM_API_KEY=sk-...
export LLM_MODEL_VERIFIER_SECURITY=deepseek-reasoner       # 只这一档升档
export LLM_MODEL_EXPLORER=deepseek-chat                    # 探查用便宜的
```

## 4. 各厂商"最佳分层组合"建议

省钱 + 提质两手都要的组合:

### DeepSeek-only(国内首选,单家便宜又强)
```
LLM_PROVIDER=deepseek
LLM_API_KEY=sk-...
LLM_MODEL_EXPLORER=deepseek-chat
LLM_MODEL_IMPLEMENTER=deepseek-coder
LLM_MODEL_VERIFIER_QUALITY=deepseek-chat
LLM_MODEL_VERIFIER_SECURITY=deepseek-reasoner   # 推理强档
LLM_MODEL_VERIFIER_DEPENDENCY=deepseek-chat
LLM_MODEL_TRIAGE_SCORER=deepseek-chat
LLM_MODEL_CHECKER=deepseek-chat                 # 与 implementer 不同模型即可
```

### Anthropic 旗舰(海外,质量优先)
```
LLM_PROVIDER=anthropic
LLM_API_KEY=sk-ant-...
# 可按需设置 LLM_MODEL_*;Codex custom agents 默认继承 Codex 会话模型
```

### OpenAI 标配
```
LLM_PROVIDER=openai
LLM_API_KEY=sk-...
LLM_MODEL_EXPLORER=gpt-4o-mini
LLM_MODEL_IMPLEMENTER=gpt-4o
LLM_MODEL_VERIFIER_QUALITY=gpt-4o-mini
LLM_MODEL_VERIFIER_SECURITY=o1                  # 推理强档
LLM_MODEL_VERIFIER_DEPENDENCY=gpt-4o-mini
LLM_MODEL_TRIAGE_SCORER=gpt-4o-mini
LLM_MODEL_CHECKER=gpt-4o-mini
```

### Qwen 国内 + 阿里云生态
```
LLM_PROVIDER=qwen
LLM_API_KEY=sk-...
LLM_MODEL_IMPLEMENTER=qwen-coder-plus
LLM_MODEL_VERIFIER_SECURITY=qwen-max            # 旗舰档
LLM_MODEL_EXPLORER=qwen-turbo                   # 探查便宜
```

### 跨厂商组合(更激进,maker 用 A 家 checker 用 B 家)
```
# 默认走 DeepSeek
LLM_PROVIDER=deepseek
LLM_API_KEY=sk-deepseek-...
# 但 checker 强制换厂商:在 goal_loop 里把 IMPLEMENTER 留 DeepSeek、CHECKER 跨到 Kimi
# (需要在你的执行入口里二次 export,或自己加一层 _adapters 的 per-call override)
```

## 6. 切厂商时唯一需要注意的两件事

1. **`.codex/agents/*.toml` 不再保存 provider fallback**;脚本/workflow 的 LLM provider 由 env 里的 `LLM_PROVIDER` 决定。
   所以你不必去改 9 个 TOML;改 GitHub Vars/Secrets 就行。若想固定 Codex custom agent 模型,可在对应 TOML 里添加 `model = "..."`。

2. **Token 价格表只内置了 Claude 三档**(`scripts/token_report.py` 里的 `PRICING_PER_M`)。
   换厂商后`token_report.py` 仍能跑(它会按 Sonnet 价做粗估),但若要准确账单,把对应价表 append 进去:
   ```python
   PRICING_PER_M["deepseek-chat"]   = (0.07, 0.27)    # 单位 USD/M tokens
   PRICING_PER_M["qwen-plus"]       = (0.5,  1.5)
   PRICING_PER_M["moonshot-v1-32k"] = (1.5,  4.5)
   ```
   厂商定价改了就改这表,业务代码不动。

## 7. 反模式

- ❌ 在多处复制 LLM 调用代码——一定要全部走 `ModelAdapter.summarize`,否则切厂商时漏改
- ❌ 把 API key 写进代码或 commit 进仓库——只用 GitHub Secrets / env
- ❌ 用同一个厂商同一个模型同时跑 implementer 与 checker——这违反 maker/checker 分工原则
- ❌ 把 `LLM_PROVIDER` 写到 `.codex/agents/*.toml` 里——provider 属于脚本/workflow env,custom agent TOML 只管 Codex agent 行为

## 8. 一张图:LLM 调用解析顺序

```
ModelAdapter.summarize(prompt, model=None, role="verifier-security")
         │
         ▼
1. 取 provider = env.LLM_PROVIDER  或  默认 "anthropic"
2. 取 model    = arg.model  > env.LLM_MODEL_<ROLE>  > env.LLM_MODEL  > PROVIDER_PRESETS 默认
3. 取 base_url = env.LLM_BASE_URL  > PROVIDER_PRESETS 默认
4. 取 api_key  = env.LLM_API_KEY  > env.<PROVIDER>_API_KEY  > env.ANTHROPIC_API_KEY (兼容)
         │
         ▼
   provider == "anthropic" ? → 走 anthropic SDK
                            └→ 否则走 openai SDK 兼容协议(base_url + key)
         │
         ▼
   记 token 到 state/token-usage.jsonl,返回文本
```
